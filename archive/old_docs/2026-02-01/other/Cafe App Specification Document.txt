CafeOS Technical Specification: A Comprehensive Product Architecture for a Next-Generation Cafe Mobile Application
1. Executive Summary and Architectural Vision
This document articulates the complete technical and product specification for CafeOS, a mobile application designed to serve as the digital operating system for a premium cafe network. Unlike generic white-label solutions or feature-bloated super-apps, CafeOS is architected with a philosophy of "utilitarian elegance." The primary objective is to reduce the friction between desire (caffeine/sustenance) and fulfillment, while layering in a nuanced, privacy-conscious social texture and a transparent loyalty mechanism.
The overarching design ethos leverages the "Editorial" trend in mobile UI, prioritizing typography, whitespace, and calm interaction models over high-octane gamification.1 This approach aligns with a user base that values clarity and efficiency. Technically, the system is designed as an iOS-first (but platform-agnostic logic) application, relying on a robust RESTful backend, strict transactional integrity for financial operations, and an optimistic UI pattern on the client side to ensure perceived latency remains near zero.3
The scope of this specification encompasses four distinct but interconnected domains: a multi-modal Ordering Engine (Dine-in, Take-away, Delivery), a transparent Loyalty Protocol based on RFM (Recency, Frequency, Monetary) analysis, a Store Intelligence module handling geolocation and operational status, and a minimal Social Connection layer designed to facilitate real-world meetups rather than digital retention.
This is a build-ready document. It moves beyond conceptual broad strokes to define data schemas, strict user flows, edge-case handling, and the specific design tokens required for implementation by a dedicated engineering team.

2. Core Product Mechanics and Functional Pillars
The application’s functionality is strictly bounded to prevent scope creep, ensuring that every feature serves a tangible utility. The four core mechanics are defined below with their specific technical constraints.
2.1 The Ordering Engine
The ordering mechanism is the central nervous system of the application. It is not merely a form for data entry but a state machine that manages inventory reservation, price validation, and fulfillment logistics. The engine supports three distinct fulfillment modes, each triggering unique validation logic at the API level:
* Dine-in: This mode creates a digital-to-physical bridge. It necessitates the input of a TableNumber (scanned via QR code or manually entered). Crucially, this mode bypasses packaging fees and delivery surcharges. The inventory check is strictly scoped to the local store’s real-time stock, preventing orders for items that physically exist in the catalog but are momentarily depleted at the specific location.
* Take-away: This mode introduces temporal logic. It requires a PickupTime slot selection. To prevent kitchen overload, the backend implements "Throttle Balancing," limiting the number of orders acceptable per 15-minute window based on the store's throughput capacity.5 This prevents the common failure state where a digital order queue overwhelms the physical barista team.
* Delivery: This mode extends the domain beyond the store’s walls. It requires DeliveryAddress validation against a geo-fenced polygon defining the store's delivery radius. It also triggers the calculation of dynamic fees and an estimated time of arrival (ETA) that synthesizes driver availability (via third-party logistics APIs) and kitchen prep time.6
2.2 The Transparent Loyalty System
The loyalty mechanics reject the opaque "black box" algorithms often found in modern apps. Instead, CafeOS implements a transparent "Earn & Burn" model. This builds trust by making the value exchange explicit.
* Accrual Logic: Users earn a fixed rate (e.g., 10 Points per currency unit). Points are accrued only upon the COMPLETED state of an order, preventing fraud from cancelled or refunded transactions.
* Redemption Mechanics: The system treats Rewards as a distinct inventory class. A user exchanges points for a RewardObject (e.g., "Free Pastry"), which acts as a 100% discount voucher applied to a specific SKU.
* Voucher Architecture: The system supports a flexible voucher engine capable of handling percentage-based discounts, fixed-value deductions, and conditional logic (e.g., "Buy X, Get Y"). These are validated server-side to prevent stacking exploits.
2.3 Store Intelligence and Location Services
This module handles the geospatial data and the operational status of the physical network.
* The Store Registry: A master database of all locations, containing immutable metadata (Latitude/Longitude, Address) and mutable status flags.
* Dynamic Status Logic: The application must handle complex status states beyond simple "Open/Closed." It includes "Busy" flags calculated via order throughput and "Delivery Unavailable" flags if the 3PL (Third Party Logistics) network is saturated.7
* Geospatial Sorting: The client utilizes the device's location services to sort stores not just by linear distance, but by "functional distance"—prioritizing open stores and those within delivery range over closer but closed locations.
2.4 Light Social Connection
The social feature is designed as a utility for coordination, not content consumption. It avoids feeds, likes, or algorithmic engagement loops.8
* Privacy-First Presence: The "Connect" feature allows users to broadcast their presence at a specific store ("I'm at Downtown Cafe") only to a mutually accepted "Inner Circle" of friends. This state has a strict Time-To-Live (TTL), automatically expiring after a set duration or when the user leaves the geofence, ensuring safety and preventing stalking risks.9
* Peer-to-Peer Gifting: The primary interaction is transactional benevolence—buying a product for a friend. This creates a "Gift Object" in the recipient's wallet, redeemable at the specific location.

4. User Flows: Detailed Narrative and Logic
The following section outlines the critical paths through the application. Each flow is detailed with entry points, step-by-step actions, and handling of edge cases.
4.1 First App Launch (Cold Start / Guest Mode)
Entry Point: The user taps the app icon for the first time.
Goal: Reduce time-to-value. Avoid "Login Walls." Allow exploration.
1. Splash & Initialization: The app displays a minimal logo on a solid background (Espresso color). In the background, the app initializes the API client and checks for a stored session token.
2. Contextual Permission Request: Instead of a generic iOS system prompt, the app displays a calm, designed modal: "CafeOS needs your location to show you the nearest coffee." Two buttons: "Allow Location" (Primary) and "Browse Manually" (Secondary).
   * Decision Point: If the user taps "Allow," the OS system prompt is triggered. If "Browse Manually," the user is taken to a city selector list.
3. The Store Feed (Home): The user lands on the main "Menu" tab.
   * State: The app is in "Guest Mode." The user can browse the full menu, see prices, and view store details.
   * Edge Case: If the user taps "Add to Cart" or "Favorites," the "Login/Signup" flow is triggered. This is "Just-in-Time" authentication.
4.2 Login / Signup Flow
Entry Point: Triggered by an action requiring identity (e.g., Checkout, Profile access).
Methodology: Phone Number (OTP) is the primary method. This serves dual purposes: it verifies identity (reducing spam orders) and provides a necessary contact method for delivery drivers.1
1. Input Screen: A clean screen asking, "What's your mobile number?" The input field auto-formats to the local region (e.g., (555) 123-4567).
2. OTP Verification: The user receives a 6-digit SMS code. On iOS, the app subscribes to the QuickType bar to suggest the code automatically.
   * Failure State: If the code is invalid/expired, shake the input field and show a specific error: "That code expired. We just sent a new one."
3. Profile Completion (New Users Only): If the phone number is new, a secondary screen asks for "First Name" and "Email" (for digital receipts).
4. Restoration: The user is returned exactly to the state they were in before the flow started (e.g., the specific Item Detail screen with their modifiers still selected).
4.3 Browsing the Menu
Design Philosophy: "Editorial." This is a visual experience.
1. Store Context Header: The top of the screen clearly states, "Ordering for:". Tapping this opens the Store Selector.
2. Category Navigation: A horizontal sticky bar lists categories (Coffee, Tea, Bakery). Tapping one smooth-scrolls the list to that section.
3. Product List: Items are presented as rows. Each row contains:
   * Thumbnail image (Left).
   * Title and short description (Center).
   * Base price (Right).
   * Visual Cue: If an item is SOLD_OUT in the backend inventory, the row is greyed out and unclickable.
4.4 Placing an Order (The "Order Engine")
This flow varies based on the Fulfillment Type selected.
Scenario A: Take-away (Standard Flow)
1. Item Configuration: User selects "Latte." A modal opens.
2. Modifier Selection:
   * The user must select a size (Required).
   * The user can select a milk (Defaults to "Whole"). Changing to "Oat" updates the displayed price button dynamically (+$0.80).
3. Cart Addition: Tapping "Add to Order" validates the selection and closes the modal.
4. Checkout Screen:
   * The user selects "Pickup" toggle.
   * The user selects "Time": "ASAP (approx 10m)" or a specific future slot.
   * The user confirms payment via Apple Pay or stored card.
   * Success: Transition to "Order Tracking" screen.
Scenario B: Delivery (Edge Cases)
1. Address Input: At Checkout, selecting "Delivery" prompts for an address.
2. Geofence Validation: The app sends the coordinate to the backend.
   * Edge Case (Out of Range): The backend returns a 400 error with type DELIVERY_UNAVAILABLE. The UI displays a friendly error: "This store doesn't deliver to that address. Try switching to our location?"
3. Fee Calculation: The backend returns an updated OrderSummary object including delivery_fee and service_fee.
Scenario C: Dine-in (QR Flow)
1. Entry: User scans a QR code on the table.
2. Deep Link: The app opens directly to the Menu for that store, with DINE_IN mode active and table_number pre-filled in the session.
3. Checkout: The "Delivery/Pickup" toggle is hidden. The header reads "Dine-in at Table 4." No packaging fees are applied.
4.5 Cart Management
Entry Point: The floating "View Order" button on the Menu screen.
1. List View: Items are listed with their full modifier strings (e.g., "Latte / Oat / No Sugar").
2. Editing:
   * Quantity: Simple Stepper (- 1 +). Reducing to 0 triggers a "Remove Item?" confirmation.
   * Modifiers: Tapping the item re-opens the configuration modal.
3. Conflict Resolution 13:
   * Scenario: A user adds a muffin. 10 minutes later, before checkout, the store runs out.
   * Handling: On entering the Cart, the app syncs with Inventory. The muffin item is visually flagged (Red text: "Sold Out"). The "Checkout" button is disabled until the conflict is resolved (item removed).
4.6 Payment and Voucher Application
Entry Point: Cart Footer.
1. Voucher Selection: User taps "Add Voucher." A modal lists available rewards (from Loyalty) and general coupons.
2. Application: User selects "$5 Off."
   * Calculation: The client sends a dry_run request to the Order API. The server responds with the new total. This prevents client-side math errors regarding tax/rounding.
3. Payment Processing:
   * The app uses the Stripe SDK (or equivalent).
   * Optimistic Success: Upon successful biometric auth (FaceID), the UI immediately shows the "Success" animation while the backend processes the webhook.
4.7 Order Tracking
Entry Point: Post-checkout or "Orders" tab.
1. State Visualization: A rigorous progress bar shows the order lifecycle:
   * Confirmed (Order received).
   * Preparing (Barista has tapped "Start" on KDS).
   * Ready (Order is on the counter).
2. Live Updates: The app uses a WebSocket connection or Silent Push Notifications to update this state in real-time.
3. Pickup Action: When state == READY, the screen displays a large "Pickup Number" (e.g., #92) for the user to show staff.
4.8 Store Discovery & Navigation
1. List View: Stores are sorted by a weighted algorithm: Status (Open > Closed) + Distance.
2. Map View: A map shows pins.
   * Innovation: The pin label isn't just the logo; it's the Wait Time (e.g., "5m", "15m"). This helps users distribute load away from busy stores.
3. Navigation: Tapping "Get Directions" opens the OS default (Apple Maps/Google Maps). We do not implement turn-by-turn navigation in-app.
4.9 Connect People (The "Social" Feature)
Entry Point: "Connect" Tab.
1. Check-In: User taps "I'm here."
   * Privacy: A modal asks, "Share with friends for 2 hours?"
2. Status Broadcasting: The user's avatar appears on their friends' "Connect" lists with the location "Downtown Cafe."
3. Interaction: A friend sees the status. They can tap "Gift a Coffee."
   * This opens a mini-menu. They pay. The user at the cafe receives a push: "Alex just bought you a coffee! Redeem it at the counter."

5. UI/UX Design System
The application adopts an "Editorial" design language. This aesthetic mimics high-end print magazines or productivity tools like Notion: heavy reliance on typography, stark black/white contrast, and generous whitespace. It avoids the "cartoony" or "colorful blob" aesthetic common in food delivery apps.
5.1 Design Philosophy
* Calmness: The interface should not scream for attention. It is a tool.
* Texture: Use of "paper" whites (slightly warm) rather than clinical blue-whites.
* Typography-First: Hierarchy is established through font size and weight, not colored boxes or drop shadows.
5.2 Color System

	5.4 Spacing and Grid System
The layout enforces a strict 8-point grid system.16 All margins and padding are multiples of 8.
* Grid Base: 8pt.
* Margin (Screen Edge): 16pt (Mobile standard) or 24pt (Editorial breathing room).
* Gutter (Card Spacing): 16pt.
* Stack Spacing:
   * Title to Subtitle: 8pt.
   * Subtitle to Content: 16pt.
   * Section to Section: 32pt or 48pt.
5.5 Component Library Specifications
Buttons:
* Primary: Primary.Espresso background, Surface.Card text. Height: 48px. Corner Radius: 8px (Soft square, avoiding the "pill" shape to look more serious/editorial).
* Secondary: Transparent background, 1px Primary.Espresso border.
* Interaction: On press, opacity reduces to 80%. No ripple effect (too "Material").
Cards (Menu Item):
* Elevation: 0. No drop shadows. Definition comes from a 1px border (#E0E0E0).
* Corner Radius: 12px.
* Layout: Image (80x80px) on the left (or right), Text vertically centered.
Tab Bar:
* Background: Translucent/Blur material (UIBlurEffect on iOS).
* Icons: Lucide style, 24x24px.
* Active State: Icon fills with Primary.Espresso. No text labels if possible, or very subtle labels.
Icon System:
* Source: Lucide Icons (Clean, consistent stroke width of 1.5px).
* Usage: Icons are functional (Cart, Profile, Back, Close). They are not decorative. We do not use icons to represent "Coffee" or "Croissant" in the menu; we use text or photography.
* Fallback: If a specific icon is missing, default to a generic shape (e.g., a circle) or text-only.
5.6 Accessibility
* Dynamic Type: All text elements must use scalable units (sp/pt) and support OS-level font scaling up to 200%. Layouts must use Auto Layout to expand vertically as text grows.
* Touch Targets: All interactive elements (Buttons, Icons) must have a minimum hit-test area of 44x44 points, even if the visual icon is smaller.

6. Detailed Feature Specification: The Cart Screen
The Cart Screen is the transactional funnel's critical point. It requires precise state management and error handling.
6.1 UI Layout
The screen is presented as a "Sheet" or a dedicated view.
* Header: "Your Order" (H2 Serif).
* Scroll View:
   * Grouped Items: Items are grouped by Store (though effectively the cart only supports one store at a time).
   * Cart Item Row:
      * Left: Quantity Stepper. A distinct component [ - 2 + ].
      * Center: Item Name (Bold) + Modifier String (Caption, Grey).
      * Right: Price (e.g., "$12.50").
* Footer (Sticky):
   * Upsell: A single line text "Forgot a cookie? +$2.00" (AI driven).
   * Voucher Row: "Promo Code or Reward >" (Tapping opens selection).
   * Summary: Subtotal, Tax, Delivery Fee, Total.
   * Checkout Button: Full width, Primary.Espresso. Label: "Checkout • $15.50".
6.2 Interaction Model & Logic
* Editing Modifiers: Tapping the text of an item does not just highlight it. It opens the Item Detail Modal pre-populated with the current choices. The user can change "Oat Milk" to "Whole Milk" and tap "Update Order."
   * System Logic: This replaces the CartItem in the array and recalculates the price.
* Quantity Change:
   * Incrementing (+) triggers an Optimistic UI Update. The number changes instantly. The API call happens in the background.
   * Failure: If the API returns "Insufficient Stock," the number reverts, and a Toast error appears: "Sorry, only 2 left."
* Removal: Reducing quantity to 0 transforms the row into a "Delete" confirmation state or simply removes it (with an "Undo" snackbar).
6.3 Edge Cases and Error Handling
* Price Drift: If the price of an item changes between the time it was added and the time the cart is viewed (e.g., Happy Hour ends), the Cart API response flags the item.
   * UI: A banner appears at the top: "Note: Some prices have changed." The old price is struck through, new price shown.
* Stock Depletion: If an item goes OOS while in the cart, the item row turns semi-transparent. The "Checkout" button is disabled. An error message appears inline: "Currently Unavailable." The user must remove it to proceed.

7. Feature Specification: Store & Map Experience
7.1 Store List Sorting Logic
The "Find a Cafe" list is not a static database dump. It utilizes a weighted sorting algorithm:
1. Filter: Active == True.
2. Primary Sort: Functional Availability.
   * Bucket A: Open Now AND Within Delivery Radius (if mode is delivery) OR Within 5km (if mode is pickup).
   * Bucket B: Open Now but distant.
   * Bucket C: Closed.
3. Secondary Sort: Linear Distance (Haversine formula) from User Location.
7.2 Store Detail Screen
* Header: Hero image of the specific location interior.
* Info Block: Address, Status Badge ("Open until 9pm"), Distance ("0.5mi away").
* Features: A row of icons/badges: "WiFi", "Outdoor Seating", "Drive-thru."
* Action: "Order Here" (Primary Button).
7.3 Directions & Navigation
The app creates a seamless handoff to native maps.
* Interaction: Tapping the Map Pin or the Address on the Detail screen.
* Logic: The app constructs a universal link (e.g., http://maps.apple.com/?daddr=lat,long) which opens Apple Maps on iOS and Google Maps on Android.
* Why: Building turn-by-turn navigation inside a cafe app is feature bloat and a maintenance liability.

8. Feature Specification: AI-Assisted Personalization
This feature addresses the "Paradox of Choice" and friction. It is a Context-Aware Recommender System designed to shortcut the ordering process.17
8.1 The "Quick Order" Modal
Trigger: This logic runs immediately upon App Launch (Cold Start).
The Algorithm (Simplified Rule-Based):
The system evaluates a "Context Score" for the user's history.
* Input: Current Time (e.g., 8:00 AM), Current Location, Day of Week.
* Historical Pattern: Look for orders placed at this Time +/- 1 hour and Location +/- 1km.
* Threshold: If >3 matching orders in the last 30 days are found for the same items, confidence is High.
The Experience:
* If Confidence > 80%: A modal overlays the screen.
   * Headline: "The usual, Alex?"
   * Content: "Large Oat Latte • Pickup at Downtown."
   * Action: "Order Now ($6.50)" -> Tapping this creates the Cart and deep-links directly to the Checkout Screen, bypassing the menu entirely.
* Cold Start / Low Confidence: If the user is new or history is erratic, the modal is suppressed. The Home screen instead displays "Trending at" section based on aggregate store sales data.
8.2 Privacy and Explanation
* Transparency: The modal includes a micro-copy line: "Suggested based on your weekday morning history."
* Opt-Out: The User Settings includes a toggle: "Enable Smart Suggestions."
* Failure Handling: If the user dismisses the modal (X) three times consecutively, the system enters a "Cool-down" mode and stops showing it for 7 days to avoid annoyance.

9. Feature Specification: Role-Based Access Control (RBAC)
The application code is shared, but the interface adapts based on the authenticated user's Role Claim in their JWT.19
9.1 The Customer Role (Default)
* Scope: Can READ Public Menus, WRITE Own Orders, READ Own Profile.
* UI: Standard consumer interface defined in this spec.
9.2 The Staff Role (Barista)
* Authentication: Staff login via a specific "Employee Portal" button using corporate credentials.
* UI Transformation: The "Order" tab is replaced by a KDS (Kitchen Display System) view.
   * View: A Kanban board of "Incoming," "Preparing," "Ready."
   * Capabilities: Can change Order State. Can toggle Item Availability (e.g., mark "Croissant" as OOS). This updates the global inventory instantly.
   * Restrictions: Cannot edit Menu Prices. Cannot issue Vouchers.
9.3 The Admin Role (Manager)
* Scope: Full access.
* UI Transformation: Adds a "Dashboard" tab.
   * View: Real-time sales graph, User analytics.
   * Capabilities: Can issue refund authorizations. Can edit Store Hours.

10. Non-Goals, Risks, and Trade-offs
10.1 Non-Goals
* Table Reservations: We explicitly exclude table booking logic. This adds massive complexity (time slot management, table inventory) and is ill-suited for a high-turnover fast-casual cafe model.
* In-App Chat: We will not build a chat system between users or staff. Communication is handled via automated status updates and transactional SMS.
* Driver App: We assume delivery fulfillment is handled either via a 3rd-party API integration (e.g., Uber Direct, DoorDash Drive) or that drivers use a separate, dedicated operational app. This spec covers the Consumer app only.
10.2 Technical Trade-offs
* Server-Side Cart vs. Client-Side: We choose Server-Side.
   * Trade-off: Requires a network call for every quantity change (latency).
   * Benefit: Persistence (cart saves across devices) and Inventory Truth (we validate stock immediately).
   * Mitigation: Use "Optimistic UI" patterns to mask the network latency.
* Approximate vs. Exact Location Sharing: For the social feature, we choose Approximate.
   * Trade-off: You can't see exactly which table a friend is at.
   * Benefit: Massive gain in user trust and privacy safety.

11. Conclusion
This specification defines a mobile application that is robust, scalable, and focused on high-value utility. By strictly bounding the feature set to Ordering, Loyalty, and Contextual Intelligence, CafeOS avoids the bloat that plagues modern "Super Apps." The architectural decisions—specifically the use of Server-Side state management for financial integrity combined with Optimistic UI for perceived speed—ensure a product that is both reliable for the business and delightful for the user. The detailed data models and design tokens provided herein serve as the immediate blueprint for engineering and design teams to commence the build phase.
